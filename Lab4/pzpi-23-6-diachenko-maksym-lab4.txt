2 ЗАВДАННЯ
Метою лабораторної роботи є демонстрація способу масштабування backend-рівня програмної системи для роботи з великим навантаженням. У межах роботи потрібно описати стратегію масштабування, технічні рішення, що роблять масштабування можливим, визначити вузькі місця та підготувати матеріали навантажувального тестування.

3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему та backend-рівень
Програмна система аналізу природних ресурсів призначена для збирання, зберігання, перевірки та аналізу даних про земельні, водні й лісові ресурси. Backend-рівень системи забезпечує REST API для web- і mobile-клієнтів, авторизацію користувачів, доступ до довідників, обробку звітів, імпорт даних та експорт аналітики.

3.2 Початкова архітектура backend
Початковий backend складається з API-сервера, бази даних PostgreSQL і кешу Redis. API-сервер обробляє HTTP-запити, виконує бізнес-логіку та звертається до бази даних. У простому варіанті один сервер може обробляти невелику кількість користувачів, але при збільшенні кількості одночасних запитів він стає вузьким місцем.

3.3 Обрана стратегія масштабування
Для лабораторної роботи обрано горизонтальне масштабування. Це означає, що замість збільшення потужності одного сервера запускається кілька однакових копій API-сервера. Вхідний трафік розподіляється між ними через load balancer. Такий підхід добре підходить для REST API, якщо сервери є stateless і не зберігають критичний стан у локальній памʼяті.

3.4 Технічні рішення, що роблять масштабування можливим
Масштабування backend забезпечується такими рішеннями: stateless API, винесення стану до PostgreSQL і Redis, контейнеризація Docker, балансування через Nginx, Kubernetes Deployment для запуску реплік, Kubernetes Service для доступу до сервісу, Horizontal Pod Autoscaler для автоматичного збільшення кількості pod-ів, readiness/liveness probes для контролю стану контейнерів.

3.5 Компонентна схема масштабованого backend
Масштабований backend складається з клієнтів, балансувальника навантаження, групи API pod-ів, PostgreSQL, Redis та модуля навантажувального тестування Locust. Запити від клієнтів надходять до load balancer, після чого розподіляються між активними копіями API. API-сервери звертаються до бази даних для постійного зберігання й до Redis для кешування часто використовуваних даних.

3.6 Docker Compose конфігурація
Для локальної демонстрації підготовлено docker-compose.yml. Він запускає Nginx, кілька копій API, PostgreSQL та Redis. За допомогою параметра --scale api=3 можна перевірити роботу кількох реплік API-сервера. Nginx приймає запити на порті 8080 і перенаправляє їх до API.

3.7 Kubernetes конфігурація
Для демонстрації промисловішого підходу підготовлено Kubernetes manifests: namespace, deployment, service, horizontal pod autoscaler та deployment для PostgreSQL. Deployment описує контейнер API, кількість реплік, ліміти ресурсів, readinessProbe і livenessProbe. Service надає стабільну точку доступу, а HPA масштабує API за CPU utilization.

3.8 Аналіз вузьких місць
При навантаженні першими можуть вичерпуватися CPU API-сервера, пул з’єднань до PostgreSQL, пропускна здатність бази даних, памʼять контейнерів, мережеві ресурси load balancer і швидкість виконання складних аналітичних запитів. Для зменшення цих ризиків використовуються кешування, індекси БД, ліміти ресурсів, health checks, репліки API та розділення важких фонових задач.

3.9 Навантажувальне тестування
Для перевірки масштабування підготовлено Locust-сценарій. Він імітує відкриття списку ресурсів, фільтрацію за типом ресурсу, отримання підсумкового звіту та health-check. Демонстраційні результати показують, що зі збільшенням кількості реплік API зростає кількість запитів за секунду: 1 репліка — 185 RPS, 2 репліки — 348 RPS, 3 репліки — 512 RPS, 4 репліки — 665 RPS. Одночасно p95 latency зменшується з 620 мс до 330 мс.

4 ВИСНОВКИ
У результаті виконання лабораторної роботи було розроблено та описано стратегію масштабування backend-рівня програмної системи аналізу природних ресурсів. Було обрано горизонтальне масштабування API-серверів, підготовлено Docker Compose і Kubernetes конфігурації, визначено основні вузькі місця системи та створено сценарій навантажувального тестування Locust. Отримані демонстраційні результати показали, що збільшення кількості реплік API підвищує кількість запитів за секунду і зменшує затримку відповіді.
