2 ЗАВДАННЯ

Метою лабораторної роботи є розроблення мобільного застосунку, який є частиною програмної системи аналізу природних ресурсів і взаємодіє з іншими компонентами системи. Мобільний застосунок реалізовано як Android-застосунок мовою Kotlin.

3 ОПИС ПРИЙНЯТИХ ІНЖЕНЕРНИХ РІШЕНЬ

Мобільний застосунок призначений для польових операторів та аналітиків. Він забезпечує перегляд короткої аналітичної інформації, списку природних ресурсів, створення польових спостережень та синхронізацію з серверною частиною.

Обрана архітектура мобільного застосунку базується на поділі на шари:
- UI layer — екрани та компоненти інтерфейсу;
- ViewModel layer — стан екранів і події користувача;
- Repository layer — взаємодія з джерелами даних;
- Network layer — HTTP API-клієнт;
- Model layer — класи даних системи.

Такий підхід спрощує супровід, тестування та розширення застосунку. Дані зберігаються на сервері, а мобільний клієнт виступає як інтерфейс для швидкого введення та перегляду інформації.

4 ВЗАЄМОДІЯ З ІНШИМИ ЧАСТИНАМИ СИСТЕМИ

Мобільний застосунок взаємодіє із серверною частиною через REST API. Користувач створює запис спостереження, застосунок формує JSON-запит і надсилає його на сервер. Сервер перевіряє дані, зберігає їх у базі та повертає результат операції.

Основні API-операції:
- GET /api/resources — отримання списку типів ресурсів;
- GET /api/observations — отримання спостережень;
- POST /api/observations — створення нового спостереження;
- GET /api/dashboard/summary — отримання короткої аналітики.

5 UML-ДІАГРАМИ

5.1 Use Case Diagram

Актори: польовий оператор, аналітик, серверна система. Основні прецеденти: перегляд показників, створення спостереження, синхронізація даних, перегляд аналітичного підсумку.

5.2 Component Diagram

Компоненти: Mobile UI, ViewModel, Repository, API Client, Backend API, Database. Застосунок не працює напряму з базою даних, а використовує серверний API.

5.3 Activity Diagram

Послідовність створення спостереження: відкриття форми, введення даних, валідація, надсилання на сервер, отримання відповіді, оновлення списку.

5.4 State Diagram

Стан запису спостереження: Draft, ReadyToSync, Syncing, Synced, Failed. У разі помилки запис може бути повторно надісланий.

6 ВИСНОВКИ

У лабораторній роботі було спроєктовано мобільний застосунок для програмної системи аналізу природних ресурсів. Було визначено його архітектуру, взаємодію з серверною частиною та створено кодову структуру Android-застосунку мовою Kotlin.
