2 ЗАВДАННЯ

Метою лабораторної роботи є розроблення front-end частини програмної системи, яка підтримує роботу клієнтів у web-режимі та забезпечує адміністрування системи. Web-застосунок повинен мати окремий інтерфейс для звичайних користувачів і окремий інтерфейс для адміністраторів відповідно до ролей користувачів системи.

У межах роботи необхідно реалізувати підтримку інтернаціоналізації та локалізації: українську й англійську мови інтерфейсу, формат дати й часу відповідно до регіону, порядок сортування текстових значень та різні напрями введення тексту. Адміністративний інтерфейс повинен забезпечувати керування користувачами, керування даними системи, створення резервних копій, експорт та імпорт даних і налаштувань.

Звіт повинен містити опис прийнятих інженерних рішень, опис взаємодії компонентів web-застосунку між собою та з іншими елементами системи, UML-діаграму прецедентів, UML-діаграму компонентів і дві додаткові UML-діаграми на вибір. Для цієї роботи додатково побудовано UML-діаграму пакетів та UML-діаграму діяльності.

3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Загальні відомості про систему та роль web-застосунку

Проєкт має назву Natural Resources Analysis Software System, або «Програмна система аналізу природних ресурсів». Його призначення полягає у централізованому збиранні, зберіганні, валідації та аналізі даних про природні ресурси регіону. Система працює з показниками земельних, водних, лісових та інших ресурсів, дозволяє формувати аналітичні звіти, переглядати динаміку змін за періодами та експортувати результати.

Web-застосунок є основним клієнтським інтерфейсом системи. Через нього користувачі можуть переглядати аналітичну панель, фільтрувати дані за типом ресурсу, територією та періодом, переглядати таблиці показників і виконувати експорт результатів. Адміністратор через окремий режим отримує доступ до керування користувачами, довідниками, наборами даних, імпортом, експортом і резервним копіюванням.

3.2 Прийняті інженерні рішення

Для front-end частини обрано архітектуру односторінкового web-застосунку. Такий підхід дозволяє швидко перемикати режими роботи без повного перезавантаження сторінки, а також розділити логіку інтерфейсу, локалізації, стану та взаємодії з backend API.

Клієнтська частина поділяється на такі логічні модулі:
- UI Components - компоненти панелі показників, таблиці ресурсів, інструментальної панелі та адміністративного інтерфейсу;
- i18n Module - модуль перекладів, форматування дати, локалізованого сортування та керування напрямом тексту;
- API Client - рівень взаємодії з серверною частиною через REST-запити;
- State Store - локальний стан активної мови, ролі, фільтрів і завантажених даних;
- Admin Panel - адміністративні функції керування користувачами, даними, резервним копіюванням та імпортом/експортом.

У демонстраційній реалізації backend-запити змодельовано через mock API. Це дозволяє показати принципи взаємодії без необхідності розгортати повний сервер під час перегляду лабораторної роботи. При цьому структура коду залишає можливість замінити mock API на реальні HTTP-запити до backend-сервісу.

3.3 Функціональні можливості web-застосунку

Для звичайного користувача реалізуються такі можливості:
- перегляд загальної аналітичної панелі;
- перегляд кількості записів, перевірених наборів даних і середнього значення показників;
- фільтрація ресурсів за типом: усі, ліси, вода, земля;
- перегляд таблиці показників з локалізованим форматуванням дати;
- перемикання мови інтерфейсу між українською та англійською;
- експорт аналітичних даних у демонстраційному режимі.

Для адміністратора реалізуються додаткові можливості:
- перегляд списку користувачів та їхніх ролей;
- керування адміністративними діями;
- створення резервної копії;
- демонстраційний імпорт даних;
- демонстраційний експорт налаштувань і показників;
- перевірка різних напрямів введення тексту для локалізаційних сценаріїв.

3.4 UML-діаграма прецедентів

UML-діаграма прецедентів показує функції системи з погляду двох основних акторів: звичайного користувача та адміністратора. Користувач працює з аналітичною панеллю, фільтрує дані, перемикає мову інтерфейсу, переглядає таблиці та експортує результати. Адміністратор додатково керує користувачами, довідниками, імпортом, експортом і резервним копіюванням.

Розділення прецедентів за ролями дозволяє показати, що web-застосунок підтримує role-based access control на рівні інтерфейсу. У реальній системі це розділення має бути також продубльоване на backend-рівні, щоб користувач не міг виконати адміністративні дії через прямий API-запит без відповідних прав.

3.5 UML-діаграма компонентів

UML-діаграма компонентів показує взаємодію браузера, web-застосунку, backend API та бази даних. У браузері виконуються UI-компоненти, модуль локалізації, маршрутизатор ролей, сховище стану і API-клієнт. API-клієнт звертається до Auth API, Resource API, Admin API та Export API. Серверна частина працює з базою даних, де зберігаються користувачі, ресурси, показники, звіти й резервні копії.

Таке розділення дозволяє зменшити зв’язаність між компонентами. Web-застосунок не працює безпосередньо з базою даних, а використовує backend API. Це підвищує безпеку, спрощує масштабування та дозволяє одночасно підтримувати web-клієнт, мобільний застосунок і сторонні інтеграції.

3.6 UML-діаграма пакетів

UML-діаграма пакетів відображає логічну структуру front-end коду. Пакет src/ui містить компоненти відображення: DashboardView, AdminView, ResourceTable та Toolbar. Пакет src/i18n відповідає за переклади, форматування дат, локалізоване сортування й напрям тексту. Пакет src/api містить клієнти для взаємодії з backend-сервісами. Пакет src/state зберігає активну роль, фільтри та стан застосунку. Пакет src/domain містить основні предметні сутності: Resource, Indicator, User і BackupRecord.

Такий поділ робить код більш підтримуваним. Якщо потрібно змінити переклад або форматування дат, зміни вносяться до i18n-пакета. Якщо змінюється API, коригується api-пакет. Якщо додається новий елемент інтерфейсу, він реалізується в ui-пакеті без зміни доменної моделі.

3.7 UML-діаграма діяльності

UML-діаграма діяльності описує типовий сценарій роботи користувача з web-застосунком. Спочатку користувач відкриває систему, обирає мову інтерфейсу, вибирає роль і задає фільтр ресурсів. Після цього клієнтська частина отримує дані через API, будує таблицю та аналітичні картки.

Якщо активна роль є адміністративною, користувач отримує доступ до керування користувачами, довідниками, імпортом, експортом і резервним копіюванням. Якщо роль звичайна, користувач переглядає показники, таблиці та звіти без адміністративних дій. Такий сценарій відповідає вимозі мати окремий інтерфейс для звичайних користувачів і адміністраторів.

3.8 Взаємодія компонентів web-застосунку з іншими частинами системи

Web-застосунок є частиною більшої програмної системи, яка також включає backend, базу даних, мобільний застосунок та потенційні зовнішні джерела даних. Основна взаємодія відбувається через REST API.

Типовий потік даних виглядає так:
1. Користувач у браузері задає фільтр ресурсів.
2. UI-компонент передає параметри до State Store.
3. State Store ініціює запит через API Client.
4. API Client звертається до Resource API.
5. Backend отримує дані з бази даних, застосовує права доступу та повертає результат.
6. Web-застосунок локалізує дати, сортує текстові значення відповідно до мови та відображає таблицю.

Мобільний застосунок використовує ті самі backend API, тому web-клієнт і mobile-клієнт мають узгоджену модель даних. Адміністративні зміни, зроблені у web-застосунку, впливають на дані, які надалі можуть бачити мобільні користувачі або аналітики.

3.9 Інтернаціоналізація та локалізація

Інтернаціоналізація реалізована через окремий модуль перекладів. У демонстраційному коді підтримуються українська та англійська мови. Кожен текстовий елемент інтерфейсу отримується через функцію t(lang, key), що дозволяє не змішувати логіку інтерфейсу з конкретними текстами.

Форматування дат виконується через Intl.DateTimeFormat. Для української мови використовується локаль uk-UA, для англійської - en-US. Сортування текстових значень виконується через localeCompare з відповідною локаллю, тому порядок ресурсів у таблиці може коректно змінюватися залежно від активної мови.

Також передбачено перемикання напряму введення тексту через атрибут dir. Хоча основними мовами системи є українська та англійська, підтримка напряму введення демонструє готовність інтерфейсу до майбутнього розширення на мови з іншим напрямом письма.

3.10 Приклади програмної реалізації

У межах лабораторної роботи підготовлено демонстраційний front-end на JavaScript. Основні файли реалізації:
- index.html - HTML-сторінка та точка входу web-застосунку;
- src/main.js - основна логіка інтерфейсу, рендеринг панелі, таблиці й адміністративного режиму;
- src/i18n.js - переклади, форматування дати та локалізоване сортування;
- src/api.js - mock API-клієнт для демонстрації взаємодії з backend;
- src/mock-data.js - тестові дані ресурсів і користувачів;
- src/styles.css - стилізація інтерфейсу.

Код побудований так, щоб його можна було запустити локально у браузері. Для реального проєкту mock API може бути замінений на HTTP-запити до backend-сервера без зміни загальної логіки UI-компонентів.

Розроблена структура front-end частини демонструє принципи модульності, розділення відповідальностей і взаємодії клієнтської частини з backend API. Матеріали лабораторної роботи підготовлено для завантаження до GitHub-репозиторію відповідно до вимог дисципліни.
