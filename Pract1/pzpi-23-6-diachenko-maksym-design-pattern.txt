3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони проєктування
Шаблони проєктування програмного забезпечення - це типові перевірені рішення повторюваних задач, що виникають під час побудови програмних систем. Вони описують загальну структуру взаємодії класів та об’єктів і допомагають розробникам застосовувати вже відомі архітектурні підходи.
Класичні шаблони GoF поділяються на породжувальні, структурні та поведінкові. Для практичного заняття було обрано поведінковий шаблон «Стратегія», оскільки він безпосередньо пов’язаний із заміною алгоритмів та гнучкою організацією поведінки об’єктів.
3.2 Опис обраного патерна
Назва патерна: Стратегія (Strategy). Тип патерна: поведінковий шаблон проєктування.
Основне призначення патерна Strategy полягає в тому, щоб визначити сімейство алгоритмів, інкапсулювати кожний алгоритм в окремому класі та зробити ці алгоритми взаємозамінними. Такий підхід дозволяє змінювати поведінку об’єкта без зміни його основного коду.
Патерн доцільно використовувати тоді, коли одна задача може виконуватися різними способами, а конкретний алгоритм потрібно обирати або змінювати під час виконання програми. У такому разі замість великої кількості умовних операторів використовується спільний інтерфейс стратегій.
Перевагами патерна є зменшення кількості умовних операторів, покращення розширюваності системи, можливість незалежного тестування алгоритмів та дотримання принципу відкритості/закритості. До обмежень можна віднести збільшення кількості класів та необхідність правильно обирати конкретну стратегію.
3.3 Структура шаблона
Патерн Strategy складається з чотирьох основних учасників: інтерфейсу стратегії, конкретних стратегій, контексту та клієнтського коду. Інтерфейс стратегії визначає спільний метод для всіх алгоритмів. Конкретні стратегії реалізують різні варіанти поведінки. Контекст зберігає посилання на поточну стратегію та викликає її метод. Клієнтський код обирає потрібну стратегію.
3.4 Реалізація шаблона
Для демонстрації було створено приклад мовою C#. Приклад пов’язаний із темою проєкту «Програмна система аналізу природних ресурсів». У ньому різні алгоритми аналізу природних ресурсів реалізовано як окремі стратегії.
Інтерфейс IResourceAnalysisStrategy містить метод Analyze. Конкретні класи ForestAnalysisStrategy, WaterAnalysisStrategy та SoilAnalysisStrategy реалізують різні види аналізу. Клас ResourceAnalyzer виконує роль контексту: він зберігає поточну стратегію та дозволяє змінювати її через метод SetStrategy.
Такий підхід дозволяє додавати нові алгоритми аналізу без зміни класу ResourceAnalyzer. Наприклад, у майбутньому можна додати аналіз кліматичних показників, аналіз деградації земель або аналіз пожежної небезпеки як окремі стратегії.

4 ВИСНОВКИ
У результаті виконання практичного заняття було досліджено шаблон проєктування «Стратегія». Було проаналізовано його призначення, структуру, переваги, обмеження та особливості практичного застосування.
Патерн Strategy дозволяє відокремити алгоритми від класу, який їх використовує. Це підвищує гнучкість, розширюваність і підтримуваність програмної системи. Для програмної системи аналізу природних ресурсів цей патерн є доцільним, оскільки різні типи аналізу можна оформити як окремі взаємозамінні стратегії.

21      }

40  public class ResourceAnalyzer
41  {
42      private IResourceAnalysisStrategy strategy;
43  
44      public ResourceAnalyzer(IResourceAnalysisStrategy strategy)

47      }
48  
49      public void SetStrategy(IResourceAnalysisStrategy strategy)

51          this.strategy = strategy;
52      }
